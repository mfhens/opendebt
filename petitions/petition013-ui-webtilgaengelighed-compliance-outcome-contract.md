# Petition 013 Outcome Contract

## Acceptance criteria

1. Accessibility is treated as a mandatory requirement for all OpenDebt UIs.
2. Core UI flows are operable by keyboard.
3. Forms, focus handling, semantic structure, and contrast are testable accessibility qualities.
4. Accessibility verification is part of release readiness.
5. Documents and media introduced by a UI are covered by the relevant accessibility requirements.

## Definition of done

- Accessibility acceptance criteria are defined for UI work.
- Automated and manual accessibility verification are planned for UI delivery.
- Critical user journeys can be tested for keyboard and assistive-technology support.
- Every acceptance criterion is covered by at least one Gherkin scenario.

## Failure conditions

- Accessibility is deferred until after release.
- A core user journey cannot be completed without a mouse.
- Form errors or status changes are not perceivable to assistive technology.
- Inaccessible documents are published without an accessible alternative.
